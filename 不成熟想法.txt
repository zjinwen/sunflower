# 领域：
面向对象设计编程

用户 ：
包含商家 用户
手机号区分不同用户 ,因为手机号可能被重新分配，订单 服务交易等需要加上时间，手机号和日期完整确定用户

entity id 手机号 日期 昵称 可选:Email qq 微信 

用户地址：value object  和用户关联 存储在用户本地

因为去中心化,没有中心保存用户商户数据，需要用户自己备份，
备份方式：导出文件，自动保存到网盘 百度网盘 或者第三方存储，或者系统之间互相备份(可能数据量比较大，类似比特币 p2p)。
同时系统在用户丢失数据和全部数据，还可以正常工作，只是历史记录无法查看。

客户端用户数据全部保存在本地



产品 ：

外卖食物 本地产品 服务等

实体 entity  id  名称  标题 类型 标签 位置 价格 介绍 图片（可链接） 视频链接 网页显示 html

商户提供产品列表，用户根据关键字（名称 标题 介绍 ） 筛选添加（ 标签 位置 价格 ）  查询产品列表 

用户可用聚合多个商户的产品查询列表：显示可按商户分类显示，或者按照位置 价格 相关性关联。
产品热度：应为是去中心化，没有中心提供数据，数据只能有商户自己提供，或者第三方可用拦截数据分析，现实中商店买东西也不关注具体热度




订单：
entity id 用户手机号 商户手机号 订单号 产品id 价格信息 全部产品信息 图片（通过计算MD5或者不保存 ） 

图片服务：entity id data md5 日期


购买 选择服务,生成记录，保存两份客户和商户，非对称加密，
订单欺骗：近距离欺骗可能相对少些，容易解决，
如果我用别人手机号 下单如何解决：
1.下单时候通过短信验证码验证，缺点要发送一条短信，发送短信可以通过手机直接发送
  商户发送短信至用户：用户手机拦截短信或者用户自己输入，如果商户被恶意下单，发送大量订单，需要停止发送，采用下面的方式
  用户发送短信至商户：短信要收取一定费用，并且需要指定号码的手机发送，可以防止恶意发送短信下单

短信发送查询 
去中心没有第三方短信发送服务，需要一个手机设备。手机通信：长轮询检查是否有发送短信任务；拦截监听短信，如果有通过http发送到服务器



支付
通过订单短信验证，一般可以确定订单交易真实可靠，
entity id 用户手机号 商户手机号 订单号 ：
支付宝：
因为去中心化 没有中心没有公司 支付宝不知道是否可接入，个人对个人在线支付可能，至少前期是这样的情况

面对面支付: 快递送到当面交易，设置订单状态。

数字资产：token 用户通过购买数字资产，相当于充值，然后交易使用token
（似乎可以替代货币流通）
